\chapter{Design}
\label{ch:design}

This chapter describes \textit{what} the system does and \textit{why} each
high-level decision was taken.
The companion chapter on implementation (\Cref{ch:impl}) describes
\textit{how} the design is realised in code.

A few constraints shape every decision in this chapter.
The target domain (\Cref{sec:design-domain}) is narrow: monophonic
treble-clef lead sheets in the Real Book style, which frees the design
from a long tail of notation features that would otherwise consume
disproportionate effort.
No labelled corpus of real jazz lead sheets exists, which rules out
fully supervised end-to-end training from real data and pushes the
design toward synthetic supervision with domain transfer.
Finally, the project runs on a single consumer GPU within a Bachelor's
thesis time budget, which caps model capacity, training time, and the
number of architectural alternatives that can be explored empirically.

Each major decision in this chapter is argued in its own section and
paired with the alternatives that were weighed and rejected.
\Cref{tab:decision-register} collects those decisions in one place as
a roadmap; the section reference in each row points to the full
argument.

\section{Target Domain and Acceptance Criteria}
\label{sec:design-domain}

The system is designed for monophonic jazz lead sheets in the style of
\textit{The Real Book}\,\cite{kernfeldStoryFakeBooks2006}: a single
melody line in treble clef with chord symbols printed above the staff.
The notation contract is summarised in \Cref{tab:domain-contract}; it
defines both the inputs the design must handle and the inputs that
fall outside its scope.
Readers unfamiliar with any of the notation terms in the table are
directed to \Cref{app:notation-primer}.

The contract follows the genre rather than papering over missing
features. The Real Book is monophonic by construction, and the
multi-voice piano grand staff is a different genre; exotic meters and
non-treble clefs are common in printed music at large but vanishingly
rare in the corpus that motivates the project.
Extending the contract to cover the out-of-scope long tail would require
a much larger labelled corpus and a substantially more complex output
vocabulary, an open research direction rather than a Bachelor's-thesis
deliverable.

\paragraph{Quantitative acceptance criteria.}
The design must commit to falsifiable targets so that \Cref{ch:results}
can adjudicate whether they are met.
The thresholds below are calibrated against published baselines on
PrIMuS-like clean data, where end-to-end CRNN--CTC systems report
aggregate Symbol Error Rates (SER) in the
range \SIrange{1}{3}{\percent}~\cite{calvoZaragozaCameraPrimus2018,mayerPracticalEndToEnd2024}.

\begin{itemize}
    \item \textbf{Aggregate SER on the clean test split} below
          \SI{2}{\percent}, the target that
          \Cref{sec:design-eval} commits the model to.
    \item \textbf{Aggregate SER on the scanned test split} below
          \SI{3}{\percent}. The augmentation pipeline is judged a
          success if the gap to the clean number stays under
          one percentage point.
    \item \textbf{Melodic SER} (SER restricted to musically
          relevant tokens; see \Cref{sec:design-eval}) below
          \SI{0.5}{\percent} on both splits. Barlines and ties
          dominate the absolute edit count, but a downstream renderer
          can re-bar the output trivially.
    \item \textbf{Perfect-transcription rate} above \SI{60}{\percent}
          on the scanned test split: more than half of input staves
          come out token-perfect.
\end{itemize}

\section{Problem Decomposition}
\label{sec:design-pipeline}

The end-to-end transcription task is decomposed into five processing
stages plus a final serialisation step, with a clearly defined
contract between each pair.
\Cref{fig:pipeline-design} summarises the data flow, and
\Cref{tab:stage-contracts} gives the typed input/output for each stage.

\begin{enumerate}
    \item \textbf{Preprocessing} takes the input bytes (a PDF page or a
          raster image) and returns a clean, deskewed binary image of
          the page.
          This stage is purely image-processing; it is independent of any
          musical content.
    \item \textbf{Staff detection} takes the binary page and returns a
          list of detected \term{systems}, each consisting of a cropped
          staff image, the strip of pixels above it (where chord symbols
          live), and the bounding box on the page.
          This is the only stage that performs spatial layout analysis;
          everything downstream operates on one staff at a time.
    \item \textbf{Melody recognition} takes a single staff image and
          returns a flat sequence of LMX tokens describing the notes,
          rests, key, time signature, and barlines printed on that
          staff.
          This is the core machine-learning stage.
    \item \textbf{Grammar correction} takes the raw token sequence and
          returns a syntactically valid LMX sequence, repairing
          structural errors (misplaced accidentals, malformed time
          signatures, runaway measures) without inventing new musical
          content.
          This stage is deterministic, not learned.
    \item \textbf{Chord recognition} takes the chord-strip image
          associated with each staff and returns a list of jazz chord
          symbols.
          It is independent of the melody recognition stage and uses a
          separate neural network.
\end{enumerate}

\textbf{Why a five-stage decomposition.}
The chord-symbol stream and the melody stream require different
recognisers (staff-relative pattern recognition versus text-like OCR),
so they must be separable in the first place.
The grammar of the LMX output is, in turn, more easily enforced by a
deterministic validator than learned end-to-end from a finite training
set; isolating that logic in a dedicated stage produces more reliable
output without enlarging the model.
Both major surveys of the
field~\cite{calvoZaragozaUnderstandingOMR2020,shatriOPTICALMUSICRECOGNITION}
recommend modular decomposition for OMR systems that target specific
notation styles, since it lets each module be replaced or
retrained independently as the target domain evolves.

\subsection{Alternatives considered}
\label{sec:design-pipeline-alts}

Two end-to-end architectures were weighed against the five-stage
decomposition and rejected.

\paragraph{Full-page transformer (LEGATO\,\cite{yangLEGATO2025}).}
A single encoder--decoder transformer that consumes a full-page image
and emits the entire transcription as a long token sequence,
eliminating the staff-detection and grammar-fix stages and sharing
representations across the whole page.
Training such a model to competitive accuracy
requires either a corpus of full-page transcriptions (which does not
exist for the Real Book) or a synthetic generator capable of laying
out full pages with realistic chord placement, neither of which fits
the time budget.
A monolithic model is also opaque: a single
mis-prediction of a chord symbol cannot be debugged without re-running
the entire model, whereas a modular pipeline yields localised failures that the
grammar fixer can repair.

\paragraph{Tree-decoder transformer (SMT/SMT++\,\cite{riosVilaSheetMusicTransformer2024,riosVilaSMTPlusPlus2024}).}
A transformer that emits a hierarchical MusicXML-like tree, removing
the need for a separate grammar-correction step because the output is
structurally valid by construction.
SMT++ is a leading end-to-end model on polyphonic
piano music, but its advantages do not carry over here.
The target domain is monophonic, so the
structural advantage of a tree decoder over a flat CTC decoder is
small; its training recipe requires several hundred
GPU-hours of pre-training that the project's hardware cannot afford;
and the CRNN--CTC baseline on Camera-PrIMuS already
performs near the ceiling of the clean monophonic benchmark, leaving little headroom for
a more complex architecture to demonstrate gain.

\section{Staff Detection}
\label{sec:design-staffdetect}

The staff-detection stage is the segmentation front-end of the pipeline.
It consumes a binarised page and emits one cropped \term{System} record
per detected staff, each carrying the staff geometry, a tightly bounded
music-region crop, and a chord-strip crop positioned in the band
immediately above the staff.
Downstream stages never see the full page; their input contract is a
single staff at a time.

\subsection{Design choice: classical morphology}

Staff detection is implemented as a classical morphological pipeline
rather than as a learned segmenter.
The Real Book's staff geometry is regular: five thin horizontal lines,
equispaced within a tight tolerance, spanning most of the page width.
A morphological pipeline composed of a wide horizontal opening
(to isolate long horizontal ink runs), row-projection clustering
(to find line centroids), and a sliding 5-line window (to group
centroids into staves whose inter-line gaps are uniform within a
tolerance) recovers staves at high precision on this input
distribution, runs in milliseconds on CPU, and produces
deterministic outputs that can be debugged by visual inspection.

A learned detector would gain little on this domain: there is no
visual ambiguity to resolve, only routine pattern matching.
It would, however, add a second training pipeline, a second
annotation effort, and a second source of failure modes, none of which
the time budget supports without a corresponding accuracy gain to claim.

\subsection{Rejection gate}
\label{sec:design-staff-reject}

Real pages contain non-music regions (page titles, footers,
composer credits) that morphological staff detection can falsely
accept when their layout is staff-like, for instance a row of small
print that produces five collinear ink runs.
A staff mis-detected on such a region is then fed to the
CRNN, which dutifully emits a high-confidence but musically nonsense
token sequence.

The design therefore inserts a two-tier rejection gate between the
staff detector and the recogniser. A pre-CRNN tier uses cheap
geometric and OCR signals to discard obvious non-music regions before
the recogniser is invoked, and a post-CRNN tier reads the CTC mean
log-probability of the output and drops strips that fall below a
calibrated confidence threshold. A confidence override re-admits
geometry-rejected strips on which the recogniser is confident, but it
deliberately does not override the OCR-text rejection, since high CRNN
confidence on running text is a known false-positive mode.
The per-signal algorithm, the threshold, and its checkpoint-dependent
recalibration live in \Cref{ch:impl}.

\subsection{Alternatives considered}
\label{sec:design-staffdetect-alts}

The first two paragraphs below cover alternatives to the
morphological staff detector of \Cref{sec:design-staffdetect};
the last two cover earlier designs of the rejection gate of
\Cref{sec:design-staff-reject}.

\paragraph{Deep object detector (YOLO, RT-DETR, Mask~R-CNN).}
A learned staff detector trained on bounding-box annotations of staff
regions. Modern detectors give competitive segmentation on cluttered
pages~\cite{simonettaOpticalMusicRecognition2024}.
The annotation cost alone is non-trivial (hundreds to thousands of
manually labelled pages), and the gain over the morphological detector
on Real Book pages is in the noise. A learned detector becomes
attractive when the page layout is irregular (handwritten manuscripts,
historical scores), which is outside the project's scope.

\paragraph{Projection-profile slicer.}
An earlier version of this project used a horizontal-projection slicer
that cut the page at low-density rows.
Dropped after testing on Real Book pages with chord symbols printed
close to the staff: the chord text reduced the inter-staff gap and
produced under-segmented two-staff strips.
The morphological line detector is robust to chord-text density
because it operates on isolated long horizontal runs, not on the full
page projection.
The comparative literature on staff-line removal and
detection~\cite{dalitzComparativeStudyStaff2008} supports this choice:
morphological line operators consistently outperform pure
projection-profile methods on typeset music.

\paragraph{OCR-only rejection rule.}
A simpler rejection rule was tested first: any candidate strip on
which a general-purpose OCR returned non-empty text was marked
non-music, on the reasoning that page titles, composer credits, and
lyric lines all carry text where music staves do not.
Rejected as too brittle: legitimate staves frequently carry text in
or adjacent to the strip (rehearsal letters, dynamics, chord symbols,
expressive cues), and the binary verdict rejected them with no
recourse.
The current design preserves the original intent by demoting OCR
text density to one signal among several and adding the
CTC-confidence override for the cases where the recogniser
confidently disagrees with the geometric heuristics.

\paragraph{Learned binary classifier on staff crops.}
A small classifier trained to discriminate music from non-music
staff crops was also tried, on the assumption that the visual
difference between the two would be obvious enough for a
low-capacity model to capture from a modest labelled set.
Rejected because it did not consistently improve over the geometric
and OCR-density gates: training such a classifier required
hand-labelled non-music crops covering the actual failure
distribution (composer credits, footers with collinear text,
watermarks, multi-line title headings), and the classifier trained
on what could be labelled at low cost accepted the same
false-positive regions the heuristics already accepted.
The labelling cost was disproportionate to the marginal benefit
over a CTC-confidence post-gate that requires no additional
supervision.

\section{Data Strategy}
\label{sec:design-data}

The absence of a labelled corpus of jazz lead sheets is the dominant
constraint on this project.
The data strategy is therefore built on two complementary ideas:
\textit{synthetic supervision} for breadth and \textit{domain
augmentation} for realism.

\subsection{Synthetic supervision from PrIMuS}

PrIMuS~\cite{PrIMuSDataset} provides roughly 87\,000 monophonic music
incipits, each accompanied by ground-truth symbolic annotations
(clef, key, time signature, and the sequence of notes and rests on the
staff).
The corpus is distributed with renderings in the Sibelius engraving
style, which does not match the Real Book aesthetic that the system
must face at inference time.
The chosen strategy keeps the \textit{semantic content} of every
PrIMuS sample, discards the supplied renderings, and re-renders each
sample in the LilyJAZZ engraving style
(\Cref{fig:primus-vs-lilyjazz}).
This yields a corpus that is large enough to train a deep model, free
of manual annotation cost, and visually aligned with the target domain.

\begin{figure}[t]
    \centering
    \includegraphics[width=0.85\textwidth]{figures/fig_primus_vs_lilyjazz}
    \caption{The same PrIMuS incipit, rendered first in the original
    Sibelius engraving style supplied with the dataset (top) and then
    re-rendered in the LilyJAZZ style used for training (bottom). The
    symbolic content is identical; only the engraving style differs.
    Re-rendering aligns the synthetic training distribution with the
    Real Book aesthetic the model must face at inference time.}
    \label{fig:primus-vs-lilyjazz}
\end{figure}

Per-sample rendering variations sample the natural visual variability
of real lead sheets without altering the musical content: the time
signature is rendered with non-trivial probability as the C or cut-time
glyph rather than numerically; cautionary accidentals are added on a
fraction of samples; the global staff size is drawn from a small range
of point sizes, varying note size and horizontal density; and unbeamed
eighth notes appear in a minority of samples to expose the model to
both beamed and unbeamed groupings.
The specific probabilities and ranges are hyperparameters and live in
\Cref{ch:impl}; the design commitment is that these variations are
independent random draws applied at render time, not class-conditioned
augmentations.

\subsubsection*{Alternatives considered}

Several other public OMR corpora were surveyed before settling on
PrIMuS.
Each carries annotations that could in principle support an
end-to-end recogniser, but each fails on its visual distribution,
its voicing assumption, or its annotation format.

\paragraph{OLiMPiC.}
OLiMPiC~\cite{mayerPracticalEndToEnd2024} is the corpus that
introduced the very LMX output format adopted by this thesis
(\Cref{sec:lmx-format}), and is in that sense the closest analogue to
PrIMuS in terms of output representation.
Rejected because its content is \textit{pianoform} (grand staff,
multi-voice), whereas the system targets monophonic single-staff
input.
Training on pianoform examples would either force the recogniser to
consume capacity on visual patterns it must reject at inference, or
require an aggressive sub-selection that defeats the size advantage
of the corpus.

\paragraph{MUSCIMA++.}
MUSCIMA++~\cite{hajicMuscimaPP2017} is the most-cited handwritten-OMR
corpus and the natural reference for symbol-level supervision.
Its visual distribution is handwritten music on staff paper, whereas
the inference distribution is typeset jazz lead sheets, so the
engraving-style gap runs in the wrong direction. Its ground truth is
also a music-notation graph (MuNG) rather than a flat token sequence,
which does not match a CTC-style target without an additional
sequentialisation step that would introduce its own errors.

\paragraph{DeepScoresV2.}
DeepScoresV2~\cite{tuggenerDeepScoresV22020} is the largest synthetic
OMR corpus by image count and is the canonical reference for
object-detection-style approaches.
Rejected because its primary annotations are oriented bounding boxes
for individual symbols, supporting symbol-classification and
object-detection pipelines rather than sequence-to-sequence learning.
Converting OBB annotations into a flat token sequence is possible but
is itself a non-trivial dataset-conversion project, and the converted
labels would still be coarser than the semantic transcriptions PrIMuS
provides natively.

\subsection{Domain augmentation: from clean renders to scanned pages}

Even after the engraving style is corrected, a model trained only on
clean LilyJAZZ renders generalises poorly to real scans: the synthetic
feature distribution sits far from that of degraded, unevenly-lit,
geometrically distorted scans, exactly the domain gap framed in
\Cref{sec:problem-formulation}.
The system therefore generates a second, augmented version of every
training image that simulates the distortions characteristic of
real-world scans: page warping and lens distortion, scan misalignment,
focus blur, exposure variation, sensor noise, uneven illumination, ink
bleed, halftone banding, and JPEG compression
(\Cref{fig:aug-gallery}).
The model is trained on the union of clean and augmented samples, with
the augmented copies sharing the same ground-truth label.
This recipe is standard across recent end-to-end OMR
systems~\cite{calvoZaragozaCameraPrimus2018,mayerPracticalEndToEnd2024,riosVilaSMTPlusPlus2024,yangLEGATO2025};
the specific list of distortions used here is calibrated against a
sample of real Real Book scans.

    \centering
    \includegraphics[width=0.7\textwidth]{figures/fig_augmentation_gallery}
    \caption{Augmentation gallery. The top strip is a clean LilyJAZZ

\subsubsection*{Alternatives considered}

\paragraph{No augmentation, larger model.}
Train on clean LilyJAZZ renders only, relying on a deeper model and
larger capacity to generalise to scans.
The synthetic-to-real gap is a feature-distribution shift, not a
capacity bottleneck: adding parameters does not move the training
distribution closer to the test distribution.

\paragraph{Learned domain transfer (CycleGAN, style transfer).}
Train an unpaired image-to-image translator (e.g.\ CycleGAN) to map
clean LilyJAZZ renders to the Real Book scan distribution.
CycleGAN training is unstable and would consume a substantial fraction
of the project's GPU budget, and the mapping that minimises a cycle
loss need not preserve the structural details (notehead position,
accidental placement) that the recogniser must learn. A hand-crafted
augmentation pipeline, by contrast, gives explicit control over each
distortion's intensity, which is easier to debug when a particular
failure mode appears at evaluation time.

\paragraph{Real-only training (no synthetic).}
Hand-label enough real Real Book pages to train from scratch.
The cost rules this out: producing a fully labelled corpus of even a
few thousand staves is well outside the time budget, and the resulting
dataset would still be smaller than PrIMuS by orders of magnitude.

\subsection{Domain restriction: filtering the training set}

PrIMuS contains many samples that lie outside the lead-sheet domain:
multi-staff renders, non-treble clefs, exotic time signatures.
Three filters applied at dataset construction time restrict
training to the in-scope distribution, dropping multi-staff samples,
non-treble clefs, and time signatures outside the eight-element jazz
common-time set of \Cref{tab:domain-contract}.
Disabling them would only force the model to spend capacity on visual
patterns that never appear at inference, with no gain on the target
distribution. The same contract is enforced at inference time by the
grammar fixer (\Cref{sec:design-grammar}).
No alternative filter set was weighed at design time: the Real Book
domain fixes the contract, and the filters merely transcribe it into a
dataset operation.

\subsection{Fine-tuning on real pages}

The synthetic-to-real gap cannot be fully closed by augmentation alone:
the visual distribution of real Real Book scans has detail (specific
ink-pen artefacts, paper grain, page-layout idiosyncrasies) that no
practical augmentation pipeline can reproduce exactly.
The design therefore plans for a second training phase. After the
synthetic model has converged, a small set of manually annotated Real
Book pages is injected into the training split (only) and the model is
fine-tuned on the combined corpus.
The fine-tuning stage is lightweight by design: its purpose is to
adjust the feature distribution rather than to teach the model new
musical concepts.

\section{Output Representation: LMX}
\label{sec:lmx-format}

The model's output is a flat sequence of \term{Linearised MusicXML}
(LMX) tokens. LMX is the symbolic encoding introduced by
\textcite{mayerPracticalEndToEnd2024} as part of the OLIMPIC
benchmark and maintained as an open
specification~\cite{OMRResearchLmx2025}: a depth-first traversal of
the MusicXML score tree emits one whitespace-separated token per leaf
event, yielding a representation that is round-trippable to MusicXML
yet compact enough to serve as a CTC target.
This thesis adopts LMX rather than reinventing a project-specific
format, restricting its grammar to the subset needed for monophonic
jazz lead sheets (no tuplets, no slurs, single accidentals only,
treble clef only).
The rest of this section defines that subset and justifies the
encoding choices that interact with the model.

\subsection{Grammar}

A valid LMX sequence is a flat run of measures, each opened by a
\code{measure} token and optionally carrying a clef, key, or time
declaration before its content; within a measure the content is a flat
sequence of notes and rests, with no nesting or polyphony.
A note token gives a pitch letter and octave, then a duration and any
augmentation dots, then up to three optional trailing markers: an
accidental, a \code{tied:stop} closing an earlier \code{tied:start},
and a \code{fermata} hold attached to the preceding event. The full
production rules are given in \Cref{ch:impl}.
\Cref{fig:lmx-example} shows a rendered staff paired with its LMX
token stream.

Three encoding choices in this grammar deserve specific justification.

\textbf{Three-token time signatures.}
The time signature is emitted as three separate tokens
(\code{time}, \code{beats:n}, \code{beat-type:d}) rather than a single
joint token \code{time:n/d}.
This factorisation reduces the vocabulary size from forty-plus possible
combinations to roughly a dozen \code{beats:} and \code{beat-type:}
values, and it lets the model share representations between related
time signatures ($4/4$ and $2/4$ share \code{beat-type:4}; $6/8$ and
$12/8$ share \code{beat-type:8}).

\textbf{Bare accidental tokens.}
Accidentals appear as bare words (\code{flat}, \code{sharp},
\code{natural}) rather than prefixed (\code{acc:flat}).
This matches the OLIMPIC LMX
convention~\cite{mayerPracticalEndToEnd2024} and keeps the token
sequence compact.
Accidentals appear \textit{after} the duration in the token stream
because that mirrors the temporal order in which a reader processes a
note: pitch position, then duration, then any required display marks.

\textbf{Display-only accidentals.}
An accidental token (\code{flat}, \code{sharp}, \code{natural}) signals
that a flat, sharp, or natural sign should be \textit{drawn} on the
staff; it does not encode the pitch's chromatic alteration.
The chromatic pitch is already determined by the pitch letter, the
octave, and the active key signature.
This is the rule used by MusicXML, and it makes the model's prediction
task more local: deciding whether to display an accidental depends
only on the current measure and the key signature, not on a global
pitch inference.
A consequence is that the converter must compute, per note, whether
the pitch's alteration conflicts with the active key signature or with
a prior alteration in the same measure, and emit an accidental token
only when a sign should be drawn.

\subsection{Vocabulary}

The model treats LMX tokens as opaque class labels, with the three
special tokens (CTC blank, padding, out-of-vocabulary) reserved at
fixed low indices and the remaining LMX tokens assigned in a stable
order. The one design decision worth recording is that the full pitch
and octave ranges are reserved even when absent from the corpus, so
that an out-of-distribution pitch prediction decodes gracefully rather
than collapsing to \code{<unk>}. The concrete index assignment is
deferred to \Cref{ch:impl}.

\subsection{Alternatives considered}
\label{sec:design-lmx-alts}

\paragraph{MusicXML.}
The de-facto interchange format for score data.
As a direct model output MusicXML is too verbose: each note is
wrapped in nested \code{<note>$\langle$pitch$\rangle$\dots} tags that
contribute large numbers of tokens with no musical content.
MusicXML is instead the downstream target. LMX is round-trippable to
MusicXML through a deterministic post-processor, isolating the
model's learning task from the XML formatting overhead.

\paragraph{PrIMuS \code{.semantic} encoding.}
The native PrIMuS textual encoding.
It is close to LMX in spirit but encodes pitch and accidentals
jointly (\code{note-Bb5\_quarter} rather than a flat sequence) and uses
a project-internal vocabulary not adopted by downstream tooling.
LMX is more compact, more easily round-trippable to MusicXML, and
matches the OLIMPIC reference work, keeping the present results
directly comparable.

\paragraph{\code{**kern} (Humdrum).}
A compact, pipe-delimited format widely used in musicology
research~\cite{huronHumdrum2002,Humdrum}.
Its tooling ecosystem outside musicology is thin, and its expression
of accidentals and rhythm requires per-cell escape characters that
complicate CTC training.

\paragraph{ABC notation.}
A line-oriented format originally designed for folk-tune
exchange~\cite{walshawABC2011}.
The same reasoning applies as for \code{**kern}: tooling support
outside the folk-music community is limited, and its spacing and
headers are stylistically opinionated in a way that LMX's flat token
stream avoids.

\section{Model Design}
\label{sec:design-model}

The melody recognition stage instantiates the canonical CRNN--CTC
pipeline surveyed in \Cref{sec:soa-crnn} and illustrated in
\Cref{fig:soa-crnn-pipeline}.
This section justifies the choice of that architectural family and
its specific instantiation here; implementation details (module
structure, exact stride table, optimiser state) live in
\Cref{ch:impl}.

\subsection{Why CRNN--CTC}

A single staff from a jazz lead sheet is read strictly left to right
and produces a variable-length token sequence shorter than the image
is wide. This is the regime where
CTC~\cite{gravesConnectionistTemporal2006} matches the data
assumptions exactly, learning the alignment between input frames and
output tokens without per-frame supervision.
\Cref{fig:ctc-alignment} illustrates the alignment mechanism on a
three-token example.
The full literature trail behind this choice is laid out in
\Cref{sec:soa-crnn}: end-to-end CRNN--CTC has been the strongest
baseline for monophonic OMR since
\textcite{calvoZaragozaEndToEndNeural2018} and remains so for the
present
domain~\cite{calvoZaragozaCameraPrimus2018,mayerPracticalEndToEnd2024}.

\input{figures/fig_ctc_alignment}

\subsection{Alternatives considered}
\label{sec:design-model-alts}

\paragraph{Encoder--decoder transformer with attention (TrOMR-style).}
A transformer encoder that ingests the staff image as a sequence of
patches and a transformer decoder that emits LMX tokens
auto-regressively, supervised by cross-entropy on the token
sequence~\cite{riosVilaSheetMusicTransformer2024}.
Transformer decoders need substantially more training data than
CRNN--CTC to reach comparable accuracy on monophonic input, which the
project cannot afford to generate. The autoregressive decoder also
amplifies exposure-bias errors: once a single token is mispredicted,
subsequent tokens are conditioned on the wrong context, a failure mode
that CTC's frame-independent prediction avoids. And on monophonic Real
Book staves the input--output alignment is nearly monotonic and
length-bounded, the regime where CTC is known to match or exceed
attention-based decoders at a fraction of the compute cost
(\Cref{sec:soa-transformer}).

\paragraph{Tree-decoder transformer (SMT++\,\cite{riosVilaSMTPlusPlus2024}).}
A transformer that emits a structured tree representing the score's
hierarchical syntax, removing the need for a post-hoc grammar fix.
Rejected because the structural advantage of tree decoders manifests
primarily on polyphonic, multi-voice scores; on a monophonic flat
token stream the gain over CTC + deterministic grammar fix is small,
while the training cost and architectural complexity are large.

\paragraph{Pure-CNN classifier (no recurrence).}
Replace the BiLSTM with a deeper CNN and treat each column-of-features
as an independent class prediction.
Rejected because monophonic OMR requires non-trivial horizontal
context to disambiguate adjacent symbols (a dot belongs to the
preceding note; an accidental belongs to the following note; a beam
groups several notes).
A purely convolutional column classifier has only the fixed receptive
field of its filters to draw on, whereas the recurrent layer
propagates context along the whole staff; it is precisely this
limitation that motivated the recurrent stage in the CRNN family that
became standard for monophonic OMR
(\Cref{sec:soa-crnn})~\cite{calvoZaragozaEndToEndNeural2018}.

\subsection{Backbone choice}

The default backbone is a modified
ResNet18~\cite{heDeepResidualLearning2016} with asymmetric strides
that compress the height dimension aggressively while preserving
width, leaving the horizontal dimension as the CTC time axis.
ResNet18 was preferred for two reasons.
First, its residual connections give the gradient a direct path
through the network and accelerate convergence relative to comparable
plain CNNs of the VGG family.
Second, its parameter count fits comfortably within the VRAM budget of
a single consumer GPU and leaves enough headroom for a non-trivial
batch size, which is necessary for stable CTC training
(see \Cref{ch:mgmt}).
The backbone is trained from scratch on the synthetic corpus rather
than initialised from ImageNet weights: the input is a single-channel
monochrome strip with a distribution far from natural images, and the
training corpus is large enough that random initialisation converges
in a comparable number of epochs without the channel-averaging
indirection that ImageNet weights would require.

\subsubsection*{Alternatives considered}

\paragraph{Deeper ResNets (ResNet34, ResNet50).}
Tested as ablations and rejected: the increase in parameter count
slowed each epoch noticeably without measurable SER improvement on
the validation split. The bottleneck on this task is the
synthetic-to-real domain gap, not backbone capacity.

\paragraph{EfficientNet / MobileNet.}
Lightweight backbones designed for mobile inference. Their channel
widths and depth-wise convolutions are tuned for natural-image
classification, not for a tall, narrow monochrome strip with
asymmetric aspect ratio. Preliminary tests with EfficientNet-B0
converged more slowly than ResNet18 and offered no inference-speed
advantage on the project's hardware. Not adopted.

\paragraph{Vision Transformer (ViT, Swin).}
A patch-based transformer encoder. Rejected on the same training-data
argument as the encoder--decoder transformer above: ViTs are
data-hungry and the project's synthetic corpus, while large in
absolute terms, has lower visual diversity than the natural-image
datasets ViTs are typically trained on.

\paragraph{VGG-style stack (legacy).}
A five-block VGG-style stack with progressive max-pooling is retained
in the codebase as a legacy alternative for backward compatibility
with earlier checkpoints. It converged more slowly than ResNet18 on
the same data and is not used for new experiments.

\subsection{Temporal resolution and width compression}

The backbone compresses the input width by a fixed factor, trading the
number of CTC time steps against compute: too little compression is
wasteful, while too much can leave fewer time steps than output tokens
and break the CTC length constraint. The design fixes this factor at
the smallest ratio that keeps every in-scope sample within the
constraint while leaving the time axis rich enough to separate adjacent
notes on densely engraved staves; the exact value lives in
\Cref{ch:impl}.

\subsection{Recurrent layer}

The convolutional features are fed to a stacked bidirectional
LSTM~\cite{hochreiterLongShortTermMemory1997}, since per-frame
predictions on a music staff require non-trivial horizontal context
on both sides of the current position (an accidental belongs to the
following note; a dot to the preceding one).
The depth and width of the stack are bounded by VRAM at the chosen
batch size, and dropout between layers regularises against the
distribution mismatch between synthetic training data and real-scan
inference inputs.
Concrete hidden sizes, layer counts, and dropout probabilities live
in \Cref{ch:impl}.

\subsubsection*{Alternatives considered}

\paragraph{GRU in place of LSTM.}
A gated recurrent unit uses two gates instead of the LSTM's three and
trains slightly faster with a smaller parameter count.
Not the default: the recurrent stage is not the memory bottleneck at
the chosen depth (the convolutional backbone dominates both parameters
and compute), so the saving is marginal, and the LSTM is the more
thoroughly characterised cell for CTC-based recognisers.
There was no accuracy argument for deviating from it.

\paragraph{Temporal convolution or self-attention instead of recurrence.}
Replace the BiLSTM with a temporal-convolutional stack or a
self-attention layer over the feature columns.
A temporal-convolutional network has a fixed receptive field that must
be sized in advance for the longest context dependency, which on
densely engraved staves is data-dependent and awkward to bound,
whereas the BiLSTM adapts its effective context to the input.
Self-attention, in turn, reintroduces the data-hunger of transformer
components (\Cref{sec:design-model-alts}) without a clear accuracy gain
on the length-bounded monophonic input this model sees.

\subsection{Decoding strategy}
\label{sec:design-decoder}

A trained CTC model produces a per-frame distribution over the
vocabulary; turning that into a token sequence is a separate decoding
decision.
The design commits to \textbf{greedy decoding} as the default: take
the most probable class at each frame, collapse runs of repeated
classes, and drop the blank.
Greedy decoding is linear in the number of frames and the vocabulary
size, $\mathcal{O}(T\cdot V)$, and runs in a single pass.

\textbf{Beam search} of width $k$ is reported as an ablation rather
than adopted as the default.
It keeps the $k$ best prefixes at each step, at cost
$\mathcal{O}(T\cdot V\cdot k)$, and can recover from a locally
confident but globally wrong frame.
On length-bounded monophonic input, where the alignment is nearly
monotonic and the model is already confident on most frames, the
accuracy gain over greedy is small and does not justify the
multiplicative decode cost; the measured trade-off is reported in
\Cref{ch:results}.

Two grammar-aware decoders, a grammar-constrained beam and a
weighted finite-state transducer, were also considered and rejected;
because they concern the interaction between decoding and the LMX
grammar, they are discussed together with the grammar layer in
\Cref{sec:design-grammar-alts}.
The design keeps decoding and grammar enforcement separate: the
recogniser decodes greedily, and the deterministic grammar fixer
(\Cref{sec:design-grammar}) repairs structural defects in a second
pass.

\section{Two-Stream Architecture for Lead Sheets}
\label{sec:design-twostream}

A jazz lead sheet contains two visually distinct streams: the melody
on the staff (graphical symbols positioned relative to staff lines,
clef-dependent) and the chord symbols typeset above it (short
alphanumeric strings such as \code{Cmaj7}, \code{G-7}, \code{Bb13},
read as text and independent of staff geometry).
The architectural response is a \textit{two-stream} design, summarised
in \Cref{fig:two-stream-arch}: two independent CRNN--CTC recognisers
of the same family (\Cref{sec:design-model}) but with disjoint
vocabularies and disjoint training data, merged only in
post-processing.

\input{figures/fig_two_stream}

\subsection{Why two networks rather than one}

A single network trained to emit both LMX tokens and chord characters
would have to learn two unrelated visual distributions and a vocabulary
that is the union of their token sets, with most tokens absent from any
given input region.
The two-stream design instead lets each network specialise: training
data is purely in-domain, the vocabulary is minimal, and the training
schedule can be set independently for each stream.
Both surveys of the
field~\cite{calvoZaragozaUnderstandingOMR2020,shatriOPTICALMUSICRECOGNITION}
recommend this decomposition for systems that handle text-like
adjuncts to staff notation.

\subsection{Alternatives considered}
\label{sec:design-twostream-alts}

\paragraph{Single multi-task network with shared backbone.}
One CNN backbone feeds two task-specific recurrent heads (one for LMX,
one for chord characters), trained jointly with a summed CTC loss.
Rejected because the visual features useful for chord OCR (letter
strokes, numeric glyphs) and for melody recognition (notehead
position relative to staff lines) have little overlap.  Sharing the
backbone forces the network to allocate capacity to a generic feature
representation when a specialised one would do, and the joint loss
introduces a tuning headache (weighting the two losses against each
other) that the two-stream split avoids entirely.

\paragraph{Single full-page transformer with positional decoding.}
The same family as the LEGATO baseline discussed in
\Cref{sec:design-pipeline-alts}, with a single decoder that emits both
streams interleaved.  Rejected for the same reasons stated there:
prohibitive data and compute requirements, and opaque failure modes.

\paragraph{Pipeline OCR (text detector + chord parser).}
Apply a general-purpose OCR engine (e.g.\ EasyOCR, Tesseract) to the
chord strip, then parse the output text into chord symbols.
Rejected because general-purpose OCR is trained on standard text and
mis-reads stylised jazz symbols: the \code{ø} half-diminished glyph
is read as \code{0}, \code{O}, or \code{o}, and the \code{maj} suffix
is often split across two lines. A specialised CRNN trained on
synthetic Real Book chord strips reaches substantially higher accuracy
at the cost of a one-off training effort.

\subsection{Independent training, joint inference}

The two networks are trained on disjoint datasets and produce
independent outputs.
At inference time, the staff detection stage emits both a staff image
and a chord-strip image for each detected system; the two are processed
in parallel and their outputs are joined into a single JSON segment
keyed by the system's bounding box on the page.
No cross-attention or shared representation links the two networks;
this is a deliberate simplicity choice.

\subsection{Chord vocabulary and notation conventions}

Chord symbols in the Real Book follow a stylised convention that
differs from many notation packages: minor chords are written with a
hyphen (\code{G-7} rather than \code{Gm7}), major-seventh chords with
\code{maj} (\code{Fmaj7}), and diminished with the spelled-out
\code{dim} suffix (\code{Cdim7}).
Half-diminished is the one quality the Real Book itself spells
inconsistently, between \code{-7b5}, \code{m7b5}, and the \code{ø}
glyph. Rather than treat these as three classes, the design folds them
into a single canonical token: the synthetic renderer draws all three
printed forms (weighted to their observed corpus frequencies) but
labels every one with the same \code{ø} token, so the recogniser maps
any half-diminished spelling to one symbol regardless of how a given
page draws it. A post-processor expands \code{ø} back to the printed
\code{-7b5} at serialisation time.
The recogniser predicts a small character vocabulary covering the root
letters, accidentals, quality letters, extension digits, and
separators; out-of-scope features (\code{alt}, double accidentals,
parenthesised alterations) are mapped to the nearest in-vocabulary
spelling during label canonicalisation. The exact character set is
listed in \Cref{ch:impl}.

\section{Training Strategy}
\label{sec:design-training}

\subsection{Two-phase plan}

The training procedure has two distinct phases.
\textit{Phase 1} trains the model from scratch on the synthetic
LilyJAZZ corpus, with both clean and augmented versions of every
sample included in each epoch.
\textit{Phase 2} fine-tunes the resulting checkpoint on a small
manually-annotated set of Real Book pages, injected into the training
split only.
The validation and test splits remain drawn exclusively from the
synthetic corpus throughout, so that phase-2 fine-tuning does not
contaminate the held-out evaluation.

This is a textbook synthetic-pretrain $\to$ real-fine-tune transfer
learning plan; its effectiveness is the principal experimental
question of \Cref{ch:results}.

\subsection{Loss and supervision signal}

The supervision signal is the LMX token sequence associated with each
image, and the loss is CTC. CTC provides alignment-free supervision:
it scores the target sequence against the per-frame predictions without
requiring a frame-by-frame label, which is what makes the synthetic
corpus usable as ground truth at all (the rationale developed in
\Cref{sec:design-model}). The mechanics of the loss are part of the
implementation (\Cref{ch:impl}).

\subsection{Data splits}

The corpus is partitioned deterministically into train, validation, and
test splits with a fixed seed.
The held-out fractions for validation and test are kept small but not
negligible---each comfortably exceeds the few-thousand-sample threshold
at which SER estimates stabilise to two decimal places---and the
remaining majority of the data is used for training.
The same sample identifier always falls into the same split across
runs, so successive experiments are directly comparable.
When fine-tuning data is added, it is injected only into the train
split; no fine-tuning sample ever enters validation or test.
Exact split fractions are stated in \Cref{ch:impl}.

\subsection{Header-less continuation staves}

Real Book pages span multiple staves per system, and continuation staves
typically do not reprint the clef and time signature.
To teach the recogniser to read such staves, the synthetic corpus includes
\textit{header-less} variants: a fraction of treble samples are re-rendered
with the clef and time-signature glyphs hidden and paired with a label from
which the corresponding tokens are removed.
The key signature is retained, both because continuation staves normally
reprint it and because hiding it would force explicit accidentals onto
key-altered notes and so alter the body of the label.
Because each header-less variant is a genuine render paired with its own
label, the image and label are consistent by construction.
This is preferable to cropping the header out of a full render at training
time: the header occupies only a few percent of the staff width and its
boundary cannot be located reliably from the rendered image, so a crop
desynchronises the pixels from the removed tokens.
The variants enter the training distribution as ordinary samples; the model
then learns pitch and rhythm without an explicit clef, which the
treble-only contract makes unambiguous.

\subsection{Rare-token oversampling}

Two LMX tokens (\code{tied:start} and \code{tied:stop}) are visually
subtle and statistically rare in the corpus, so the model
under-predicts them.
The design choice is to oversample at the dataset level---duplicating
samples whose label contains a rare token in the training split
only---rather than to up-weight the rare token in the loss.
Sample-level oversampling preserves the natural distribution of
context (the rare token appears in its real visual context, with the
correct surrounding tokens) and, crucially, is compatible with CTC,
which does not admit per-class weights in its standard formulation.

\subsubsection*{Alternatives considered}

\paragraph{Class-weighted CTC loss.}
Adjust the loss by a per-class weight that up-weights rare tokens.
The standard CTC formulation marginalises over alignments and does
not factorise into per-token contributions that can be weighted
independently.  Approximate per-token weighting is
possible but introduces gradient bias that is hard to characterise.

\paragraph{Focal-style CTC.}
A focal-loss variant of CTC that down-weights easy samples and lets
hard samples dominate the gradient.  Rejected because the rarity of
ties is a label-frequency issue, not a sample-difficulty issue:
focal-style weighting does not specifically target the under-represented
classes the project cares about.

\paragraph{Synthetic over-generation of rare-token samples.}
Generate additional PrIMuS-style samples that contain ties.
The generation cost rules this out: modifying the LilyPond generator's
musical content risks producing musically improbable tie patterns,
which would distort the training distribution rather than fix the
imbalance.

\section{Grammar Correction as a Design Layer}
\label{sec:design-grammar}

The CRNN model is trained to maximise the probability of the correct
token sequence; nothing in the training objective forces its output to
be a syntactically valid LMX sequence.
On most samples it is, but on dense or degraded inputs the raw
prediction can contain malformed fragments: a \code{pitch:C} not
followed by an \code{octave:} token, an accidental before the duration
of its note, a barline that arrives before the accumulated beats fill
the current measure.

A dedicated \term{grammar fixer} sits between the model and the
JSON output. Its responsibilities are:

\begin{itemize}
    \item enforce the LMX grammar laid out in
          \Cref{sec:lmx-format} (correct token order within a
          note, correct duration after \code{rest}, correct nesting of
          tied groups);
    \item normalise predictions outside the lead-sheet contract: any
          clef other than \code{clef:G2} is rewritten to \code{clef:G2}
          and time signatures outside the allowed eight-element set are
          rejected;
    \item clamp predicted octaves to the in-domain range $[3, 7]$,
          preventing rare misreads from producing unreachable pitches;
    \item count cumulative beats per measure against the active time
          signature and insert implicit barlines when the count
          overflows, producing well-formed measure boundaries even when
          the model misses a printed barline.
\end{itemize}

The grammar fixer is purely deterministic: it never invents new musical
content, only repairs structural defects in the model's output.
Keeping grammar enforcement out of the loss also keeps the model's
output legible: every token in the raw prediction is interpretable in
the same way as the corresponding ground-truth token, which simplifies
error analysis (\Cref{ch:results}).

\subsection{Alternatives considered}
\label{sec:design-grammar-alts}

\paragraph{Grammar-aware CTC beam search.}
A constrained decoder that restricts the beam to grammatically valid
prefixes at each time step, much like a language-model-shallow-fusion
ASR decoder.
Rejected because the LMX grammar is a small finite-state object whose
post-hoc enforcement is essentially free; folding it into beam search
would slow inference proportionally to the beam width without
contributing extra accuracy beyond what the deterministic post-pass
already provides.

\paragraph{Weighted finite-state transducer (WFST) decoder.}
Compose the model's per-frame distribution with an explicit WFST
representation of the LMX grammar at decode time.
Over-engineered for a vocabulary this small: the hand-written
stateful walker is shorter, easier to debug, and sufficient for the
project's grammar complexity.

\paragraph{Learned sequence-to-sequence corrector.}
Train a small transformer to map raw model output to grammatically
valid output, supervised by (noisy, grammatical) pairs.
This requires a corpus of model errors that
is only available after the model has been trained (a chicken-and-egg
problem in the design stage), and it would introduce a second
data-driven component that itself has failure modes harder to debug
than a deterministic walker.

\paragraph{End-to-end grammar-aware loss.}
Augment the CTC loss with a regulariser that penalises ungrammatical
prefixes during training.
Conceivable, but it complicates loss optimisation for marginal gain on
a vocabulary as small and well-structured as LMX, and deferring grammar
to post-processing keeps training-time concerns separate from
inference-time concerns, which simplifies analysis.

\section{Evaluation Design}
\label{sec:design-eval}

The acceptance criteria stated in \Cref{sec:design-domain} commit the
project to four numeric targets.  This section defines the metrics
that produce those numbers and the protocol under which they are
collected, so that \Cref{ch:results} can adjudicate against them
without ambiguity.

\subsection{Primary metric: Symbol Error Rate}

The primary metric is the Symbol Error Rate~\cite{SymbolErrorRate}:
the Levenshtein edit distance between the predicted token sequence and
the ground truth, normalised by the length of the ground truth.
SER is reported on both the clean and the scanned versions of the test
split, so that the contribution of the augmentation pipeline is
visible.

\subsection{Category-level error breakdown}

SER aggregates errors of very different musical significance: a missed
barline counts identically to a wrong pitch.
To make this distribution legible, the evaluation reports a
category-level breakdown of edits attributable to each LMX token
family: \code{measure} tokens, tie tokens, pitch letters, octaves,
durations, accidentals, and key/time-signature tokens.
This is the same diagnostic information that OMR-NED provides as a
single number~\cite{martinezSevillaOMRNED2025}, expressed here as a
table rather than committed to a specific normalisation.

\subsection{Melodic SER}

A complementary \textit{melodic} SER is also reported, computed on the
same edit distance after stripping \code{measure} and tie tokens.
Structural tokens dominate the absolute edit count on the current
corpus but are largely musically irrelevant once a downstream renderer
re-bars the output; the melodic SER captures how often the model gets
the actual notes right.
This is the metric that the \SI{0.5}{\percent} target in
\Cref{sec:design-domain} commits to.

\subsection{Domain-gap evaluation}

The principal experimental question of this thesis is whether
synthetic pretraining plus fine-tuning closes the gap between training
on rendered images and inference on real scans.
Two ablations support this evaluation:
(i)~the model trained only on clean LilyJAZZ renders is evaluated on
real Real Book pages, exposing the raw synthetic-to-real gap;
(ii)~the same model after fine-tuning on a small annotated real set is
evaluated on the same pages, quantifying the reduction in the gap.
The fine-tuning protocol is the same as that described in
\Cref{sec:design-training}.

\subsection{Reporting protocol}

To make the four numeric targets falsifiable, the evaluation protocol
commits in advance to:
(a)~the split definitions and seed used to construct them, fixed once
and not retuned per experiment;
(b)~the decoding strategy used to produce predictions: greedy CTC
decoding as the primary number, with beam search reported as an
ablation, as set out in \Cref{sec:design-decoder};
(c)~the metric aggregation rule (micro-averaged: total edits divided
by total ground-truth length across the split, not the mean of
per-sample SERs).
\Cref{ch:results} reports its primary numbers under this protocol;
any deviation is called out explicitly.
